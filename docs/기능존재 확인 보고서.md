Q1 : '현재 프론트는 어떤 기능들이 있는지 상세히 설명해.'

현재 프론트엔드는 지원자(Candidate)와 관리자(Admin)라는 두 가지 핵심 사용자 경험을 중심으로 설계되어 있으며, 특히 실시간 멀티모달 인터랙션과 데이터 안정성을 보장하기 위한 고급 기능들이 구현되어 있습니다.
1. 지원자(Candidate) 주요 기능
대시보드 및 공고 조회:
지원 가능한 채용 공고를 탐색하고 본인의 지원 현황을 한눈에 확인할 수 있습니다.
이력서 및 데이터 관리:
지원자는 본인의 이력서를 업로드할 수 있으며, 이 텍스트 데이터는 AI 면접관의 질문 생성(RAG)에 실시간으로 반영됩니다.
면접 전 기기 환경 점검 (Device Check):
카메라, 마이크 성능 및 브라우저 호환성을 사전에 체크하여 면접 중 발생할 수 있는 기술적 문제를 최소화합니다.
두 가지 면접 모드 지원:
텍스트 모드: 채팅 형식으로 진행되는 기본 면접 환경입니다.
비디오 모드 (멀티모달/WebRTC): 저지연 WebRTC 기술을 이용한 실시간 영상 면접입니다.
실시간 자막 (Live STT): 본인이 말하는 내용이 하단에 실시간 캡션으로 표시됩니다.
시선/감정 추적: 백엔드에서 분석된 메트릭을 바탕으로 현재 상태를 모니터링합니다.
GPU 큐 관리: 서버 부하 발생 시 countdown 배너를 통해 대기 상황을 직관적으로 안내합니다.
면접 결과 리포트:
면접 종료 후 AI가 산출한 종합 점수, 등급(A/B/C), 강점/약점 분석 및 정성적 피드백을 인터랙티브한 차트와 함께 제공합니다.
2. 관리자(Admin) 주요 기능
채용 공고 관리 (Postings CRUD):
공고를 생성, 수정, 게시할 수 있으며, 게시된 공고의 AI 평가 정책을 '동결(Freeze)'하여 평가의 형평성을 유지합니다.
지원자 모니터링 및 상세 조회:
전체 지원자 목록을 상태별(진행 중, 평가 완료 등), 날짜별로 필터링하여 관리할 수 있습니다.
특정 지원자의 상세 면접 영상, STT 기록, 감정 분석 추이 등을 타임라인 형식으로 확인합니다.
관리자 결정 오버라이드 (Decision Override):
AI의 자동 판단이 아닌 관리자가 직접 합불 상태를 최종 결정하거나 수정할 수 있는 권한을 제공합니다.
실시간 통계 대시보드 (Job Stats):
공고별 실시간 진행 지표와 멀티모달 프로젝션 데이터를 실시간 대시보드 형태로 시각화합니다.
3. 기술적 UX 편의 기능
실시간 상태 동기화 (SSE & Authority Pull):
Server-Sent Events(SSE)를 통해 페이지 새로고침 없이 면접 진행 상황이 실시간으로 동기화됩니다.
세이프 모드 (Safe Mode):
서버 연결이 불안정할 경우 UI가 '읽기 전용' 모드로 자동 전환되어 데이터 유실을 방지합니다.
블라인드 모드 (Blind Mode):
공정성을 위해 면접 질문 텍스트를 의도적으로 숨기고 음성으로만 진행할 수 있는 기능을 제공합니다.
추적 ID(Trace ID) 기반 에러 진단:
오류 발생 시 UI 상단에 트레이스 ID가 포함된 배너를 노출하여 운영자가 문제를 투명하게 추적할 수 있도록 돕습니다.
프론트엔드는 React와 Vite 기반으로 구축되어 있으며, 고급 다크 모드 스타일링과 애니메이션을 활용하여 사용자에게 프리미엄 인터랙티브 경험을 제공하도록 설계되었습니다.

Q2 : '답변 평가 기준은?'

AI 면접 시스템의 답변 평가 기준은 크게 4가지 핵심 역량(Capability) 영역으로 나뉘며, 각 영역은 구체적인 데이터 기반 규칙(Rule-based Engine)에 의해 1~5점 척도로 평가됩니다.
1. 4대 평가 역량 및 세부 기준
평가 영역	주요 측정 지표 (Metrics)	상세 내용
직무 역량 (Knowledge)	키워드 매칭, 코드 복잡도	답변 내 핵심 기술 키워드가 얼마나 포함되었는지, 코드 작성 시 논리적 복합성(AST Complexity)이 충분한지를 평가합니다.
문제 해결 (Problem Solving)	힌트 사용 횟수	문제 해결 과정에서 시스템이 제공하는 힌트나 재질문을 얼마나 적게 사용했는지를 측정합니다. (힌트 0회 시 만점)
의사소통 (Communication)	STAR 구조 준수 여부	답변이 Situation(상황), Task(과제), Action(행동), Result(결과)의 구조적 틀을 잘 갖추어 논리적으로 전달되었는지를 평가합니다.
태도/비언어 (Attitude)	시선 처리, 감정 분포	면접 중 시선이 중앙에 머문 비율(아이컨택)과 부정적인 감정(공포, 슬픔 등)이 노출된 비율을 분석하여 정서적 안정성을 평가합니다.
2. 가중치 적용 방식 (Weights)
직무의 성격에 따라 각 역량의 중요도가 다르게 반영되어 최종 점수가 산출됩니다.
개발 직군 (DEV): **직무 역량(40%)**과 **문제 해결(30%)**에 가장 높은 비중을 둡니다.
비기술 직군 (NON_TECH): **의사소통(40%)**과 문제 해결(30%) 능력을 더 중요하게 평가합니다.
태도 영역: 모든 직군에서 약 10%의 비중으로 참고 지표로 활용됩니다.
3. 평가 데이터 신뢰성 보장
멀티모달 통합: 비디오(DeepFace), 시각(MediaPipe), 음성(Parselmouth) 분석 데이터를 실시간으로 수집하여 태도 점수에 반영합니다.
동결 가중치 (Snapshot Weight): 채용 공고 게시 시점의 평가 기준이 세션 끝까지 유지되도록 '스냅샷' 기술을 사용하여 평가의 공정성을 보장합니다.
결정론적 해싱 (Deterministic Hashing): 동일한 입력 데이터에 대해 항상 동일한 평가 결과가 나오도록 입력값 해싱 검증 과정을 거칩니다.
최종적으로 이 점수들은 100점 만점으로 환산되어 A(우수), B(보통), C(보완 필요) 등의 등급과 함께 구체적인 피드백 리포트로 생성됩니다.

Q3 : '직무역량, 문제해결, 의사소통 질문개수의 비율은 조정할 수 있게 되어있나?'

현재 시스템은 직무 역량, 문제 해결, 의사소통 등 '분야별 질문 비중'을 조정할 수 있도록 설계되어 있습니다.
1. 직무/역량별 질문 비중 조정 (조정 가능)
JobPolicy(공고 정책)의 evaluation_weights 필드를 통해 각 역량 분야의 가중치를 설정할 수 있고, 백엔드는 이를 바탕으로 질문 개수를 자동으로 배분합니다.
동작 방식: 
DistributionCalculator 모듈이 설정된 가중치(예: 직무 40%, 문제해결 30%, 의사소통 20%, 태도 10%)와 전체 질문 개수를 계산하여 각 카테고리별로 몇 개의 질문을 던질지(슬롯)를 하드웨어적으로 결정합니다.
결정론적 배분: LLM이 무작위로 결정하는 것이 아니라, 설정된 가중치를 기반으로 수학적으로 계산된 개수만큼 각 분야의 질문이 생성되도록 강제합니다.
2. 면접 단계(Phase) 비중 (고정된 시퀀스)
OPENING, MAIN, FOLLOW_UP, CLOSING의 비중은 조정보다는 엄격한 시퀀스(Sequence) 정책을 따릅니다.
고정 구조: 면접의 첫 번째 질문은 항상 OPENING, 마지막 질문은 항상 CLOSING으로 고정되어 있습니다.
유동적 중간 단계: 그 사이의 모든 질문은 MAIN 단계로 채워지며, 특정 MAIN 질문에 대해 꼬리 질문이 필요하다고 판단될 경우에만 FOLLOW_UP 질문이 동적으로 추가됩니다. (현재 설정상 MAIN 질문당 최대 2개까지 가능)
전체 개수: 전체 질문 개수(total_question_limit) 자체는 조정할 수 있지만, "오프닝을 3문제 하겠다"와 같은 단계별 비중 조정은 현재 시스템 아키텍처상 허용되지 않습니다. (오프닝/클로징은 각 1개씩이 표준)
요약
분야별 비중 (Knowledge vs Comm): 조정 가능 (공고 설정의 가중치에 연동됨)
단계별 비중 (Opening vs Main): 고정됨 (시작-본론-결론의 표준 흐름 준수)
즉, 어떤 '능력'을 검증할지에 대한 질문 비율은 자유롭게 바꿀 수 있지만, 면접의 '형식'은 시스템이 정의한 표준 프로세스를 따르도록 되어 있습니다.

Q4 : '질문을 생성할 때, 임의로 고정된 질문을 삽입할 수 있게 되어있나?'

네, 임의로 고정된 질문을 삽입할 수 있는 '고정 질문(Fixed Question) 주입' 기능이 설계되어 있습니다.
이 기능은 면접의 신뢰성과 일관성을 확보하기 위해 다음과 같은 방식으로 작동합니다.
1. 작동 위치: 마지막 본론(MAIN) 슬롯
임의의 위치가 아닌, 본론(MAIN) 단계의 가장 마지막 질문으로 고정 질문을 삽입할 수 있습니다. 즉, CLOSING 질문 바로 직전에 관리자가 반드시 물어보길 원하는 질문을 배치할 수 있습니다.
2. 작동 메커니즘
우선순위: 공고 정책(JobPolicy)에 설정된 고정 질문이 최우선으로 적용됩니다.
처리 방식: 해당 순서가 되면 AI가 질문을 생성하지 않고, 저장된 텍스트를 그대로(Verbatim) 사용자에게 전달합니다.
활성화 조건: 시스템 설정 내 WIRING_FIXED_Q_ENABLED 플래그가 활성화되어 있어야 하며, 공고 생성 시 '고정 질문' 필드를 작성해야 합니다.
3. 주요 특징
LLM 우회: 이 슬롯에서는 AI의 판단을 거치지 않으므로, 면접관이 반드시 확인해야 하는 필수 문항이나 공통 질문을 안정적으로 던질 수 있습니다.
유연성: 만약 고정 질문이 설정되어 있지 않다면, 시스템은 자동으로 일반적인 AI 생성 모드(GENERATED)로 전환하여 자연스러운 면접을 이어갑니다.
요약
관리자는 마지막 본론 질문 위치에 원하는 특정 질문을 고정적으로 삽입하여, 모든 지원자에게 동일한 확인 사항을 묻는 용도로 활용할 수 있습니다.

Q5 : '공고 정책 동결(Job Policy Freeze) 및 데이터 거버넌스 설계원칙 에 관한 정보와 상세설명'

시스템은 면접의 공정성을 위해 **"AI 성능에 영향을 주는 정보는 게시 후 절대 수정할 수 없다"**는 엄격한 규칙을 따르고 있습니다. 문의하신 내용을 바탕으로 현재 시스템의 동작 방식을 정리해 드립니다.
1. 데이터 거버넌스 및 수정 가능 여부 요약
구분	주요 정보 예시	LLM 전달	수정 가능 여부	설명
① 핵심 공고 정보	직무(Title), 자격요건, 우대사항 등	일부 전달	게시 전 가능	면접 질문 생성 시 **Context(문맥)**로 활용되므로 게시 후 수정이 금지됩니다.
② AI 정책 정보	질문 수, 질문 비중, 평가 가중치, 면접 모드	직접/간접 전달	게시 후 불가 (Snapshot)	질문 개수 배분 및 평가 엔진에 즉시 반영되는 정보로, 공정성을 위해 게시 시점에 Snapshot으로 동결됩니다.
③ 관리/운영 정보	채용 인원, 근무지, 마감일 등 (Metadata)	전달 안됨	게시 후 가능	면접 로직이나 AI 성능에 영향을 주지 않는 순수 운영 정보로, 
update_metadata를 통해 언제든 수정 가능합니다.
④ 시스템 무결성 정보	공고 ID, 생성 시간, 작성자 등	전달 안됨	절대 수정 불가	시스템 추적 및 감사(Audit)를 위한 데이터로, 사용자 인터페이스를 통한 수정을 원천 차단합니다.
2. 상세 설명
A. LLM에게 필요한 정보만 전달하는가?
네, 그렇습니다. 
LLMQuestionGenerator는 질문을 생성할 때 공고의 모든 데이터를 무분별하게 보내지 않습니다.
전달되는 정보: 현재 면접 단계(Step), 이전 질문 이력(History), 그리고 핵심 직무 맥락(Resume Summary 등)을 조합하여 프롬프트를 최적화합니다.
전달되지 않는 정보: 채용 인원, 마감일, 작성자 정보 등 면접 질문의 질과 상관없는 데이터는 필터링되어 LLM에게 노출되지 않습니다.
B. 게시(Publish) 후 수정이 불가능한 이유는?
우리 시스템에는 **'Policy Snapshot'**이라는 개념이 있습니다.
공고가 DRAFT에서 PUBLISHED 상태로 변경되는 순간, AI와 관련된 모든 설정 값들이 '박제'됩니다.
이후 지원자가 면접을 시작하면 해당 시점의 스냅샷을 기반으로 세션이 생성되므로, 관리자가 중간에 가중치를 바꾸더라도 이미 진행 중인 면접의 형평성은 훼손되지 않습니다.
C. 수정 가능한 'Metadata'의 범위
시스템은 AI 성능에 영향을 주지 않는 보조 정보를 Metadata라는 별도 영역으로 관리합니다.
이 영역에 속한 정보(예: 위치, 인원 등)는 공고 게시 중에도 수정할 수 있으며, 이는 AI 질문 생성 로직과는 완전히 분리되어 작동합니다.
결론적으로, 작성해주신 표는 현재 시스템이 추구하는 **'AI 윤리 및 공정성 가드레일'**을 완벽하게 관통하는 기준입니다.

Q6 : '면접관의 페르소나를 3가지중 하나로 지정 할 수 있도록 되어있나?'

Q7 : '이력서의 모든 내용이 텍스트로 변환되었다고 가정할 때, LLM이 해당 텍스트를 기반으로 면접자의 이력서에 관한 질문을 생성하도록 설계되어있는가?'

네, 이 시스템은 이력서 텍스트를 기반으로 LLM이 맞춤형 질문을 생성하도록 설계되어 있습니다.
구체적인 구현 원리와 흐름은 다음과 같습니다:
1. 이력서 텍스트 추출 및 요약
텍스트 추출: 
IMH/api/resume.py 모듈에서 PDF나 TXT 파일이 업로드될 때 텍스트를 추출하여 resume_summary_snapshot 필드로 저장합니다.
요약 생성: 
ResumeSummarizer 객체가 추출된 원문을 기반으로 후보자의 주요 기술, 경력 수준, 배경 등을 3~5문장으로 요약합니다 
(packages/imh_providers/resume_summarizer.py).
2. LLM 질문 생성 시 컨텍스트 주입
질문을 생성하는 코어 로직 (packages/imh_providers/question.py)을 보면, 다음과 같이 요약된 이력서 내용을 LLM 프롬프트에 직접 포함시킵니다:
```python
# packages/imh_providers/question.py 내 LLMQuestionGenerator.generate_question
if resume_summary and step_type_name not in ("OPENING", "GENERAL_SMALLTALK"):
    truncated = resume_summary[:1000]
    user_prompt += f"\n\nCandidate Resume Summary (use as context for depth):\n{truncated}"
```
3. 단계별 적용 (Phased Flow)
MAIN 단계: 인터뷰의 핵심 단계인 MAIN 영역에서 이력서 정보를 활용합니다. LLM은 제공된 요약본을 참고하여 후보자의 실제 경험이나 기술 스택에 대해 **심층 질문(Deep-dive)**을 던지도록 유도됩니다.
개인화: 이 과정을 통해 "일반적인 질문"이 아닌, 후보자의 이력서에 기재된 성과나 기술에 특화된 질문이 생성됩니다.
4. 설계상 특징
GPU 효율성: 이력서 요약은 세션당 단 1회만 수행되어 GPU 부하를 최소화하며, 생성된 요약본은 세션 스냅샷에 고정되어 면접 내내 일관되게 사용됩니다.
폴백 로직: 만약 이력서가 없거나 텍스트 변환에 실패한 경우, 시스템은 "표준 지원자"로 간주하는 일반적인 폴백(Fallback) 요약을 사용하여 면접이 중단되지 않도록 보호합니다.
요약하자면, 시스템은 **[이력서 업로드] → [텍스트 추출] → [LLM 요약] → [질문 생성 프롬프트에 주입]**의 파이프라인을 통해 후보자 맞춤형 질문을 생성하도록 이미 통합되어 있습니다.

Q8 : '공고의 내용 또한 LLM에 전달되어, 공고에 맞는 면접이 진행되도록 되어있나?

구조적인 설계상으로는 공고의 내용(Job Description)을 포함하는 JobPolicy 스키마가 정의되어 있으며, 
이를 LLM에 전달할 수 있는 기반이 갖추어져 있습니다. 하지만 현재의 실제 구현 상태는 다음과 같은 특이사항이 있습니다.
1. 설계 및 기반 데이터 (Architecture & Data)
JobPolicy 모델 (packages/imh_job/models.py)에는 description (주요 업무), requirements (자격 요건), preferences (우대 사항) 필드가 이미 존재합니다.
면접 세션이 시작될 때 이 정보는 job_policy_snapshot이라는 이름으로 박제(Snapshot)되어 세션 컨텍스트에 저장됩니다.
2. 현재 구현 상태 (Current Implementation)
코드를 정밀 분석한 결과, **질문 생성 단계에서 공고 내용을 LLM에 전달하는 로직은 아직 고도화 단계(Placeholder)**에 머물러 있습니다.
현재의 프롬프트 구성: 
packages/imh_providers/question.py
를 보면, 현재는 직무 카테고리가 **"Developer"**로 고정되어 전달되고 있습니다.
```python
# packages/imh_providers/question.py
user_prompt = f"Job Category: Developer\nInterview Step: {step}\n"
```
컴포저(Composer)의 역할: 
CachedPromptComposer 등의 클래스가 프롬프트를 템플릿화하여 관리하도록 설계되어 있지만, 현재는 공고의 전체 텍스트보다는 요약된 직무명 위주로 흐름을 제어하고 있습니다.
3. 결론 및 향후 방향
설계 여부: Yes. 시스템은 공고 정보를 들고 있으며, 이를 LLM에 주입하여 "공고 맞춤형 면접"을 진행하도록 아키텍처가 잡혀 있습니다.
작동 여부: 제한적. 현재는 모든 지원자를 "Developer" 직군으로 가정하고 질문을 생성하는 기초적인 수준이며, 실제 공고의 세부 requirements를 프롬프트에 동적으로 바인딩하는 로직은 다음 고도화 작업(TASK-036 등)의 대상으로 분류되어 있습니다.
따라서 지금 당장은 공고를 다르게 올리더라도 LLM이 인식하는 기본 직무 맥락(Developer)은 동일하게 유지될 가능성이 높습니다. 
이 부분을 공고별로 특화시키려면 질문 생성 프롬프트 엔진(LLMQuestionGenerator)의 user_prompt 생성부에서 하드코딩된 부분을 정책 스냅샷 데이터로 교체하는 작업이 필요합니다.

Q9 : '적절한 질문을 생성하지 못하는 경우를 대비하여, RAG를 통해 이미 만들어진 질문들의 데이터를 사용할 수 있도록 되어있나? RAG를 통해 적절한 질문을 찾지 못했을 경우에도 대비되어있나?'

네, 이 시스템은 질문 생성의 안정성을 보장하기 위해 3단계 폴백(Fallback) 전략을 완벽하게 갖추고 있습니다.
코어 엔진(packages/imh_session/engine.py)의 구현 내용을 바탕으로 설명드리면 다음과 같습니다:
1단계: RAG 및 LLM 동적 생성 (Tier 1)
먼저 LLM이 후보자의 이력서 요약, 이전 대화 기록, 그리고 RAG(검색 증강 생성)를 통해 확보한 유사 질문 데이터를 조합하여 실시간으로 가장 적합한 질문을 생성하려고 시도합니다.
2단계: 정적 질문 은행(Static QBank) 활용 (Tier 2)
만약 LLM 응답이 지연되거나, 적절한 질문 생성에 실패하면 시스템은 즉시 이미 만들어진 질문 데이터 세트로 전환됩니다.
작동 방식: imh_qbank 패키지 내의 데이터베이스에서 현재 카테고리(예: BEHAVIORAL 등)에 맞는 질문 리스트를 가져옵니다.
중복 방지: 이전에 이미 질문했던 ID는 제외하고 남은 리스트 중에서 무작위로 하나를 선택합니다.
3단계: 초비상 긴급 질문(Emergency Fallback) (Tier 3)
정적 질문 은행마저 비어있거나 모든 질문을 다 사용해버린 최악의 상황을 대비한 하드코딩된 긴급 질문 리스트가 준비되어 있습니다.
```python
# engine.py 내 실제 코드 일부
emergencies = [
    "Please introduce yourself and your key strengths.",
    "What is your biggest weakness and how are you overcoming it?",
    "Describe a time you faced a significant challenge at work.",
    "Where do you see yourself in 5 years?"
]
```
이 3단계에서는 절대로 실패하지 않는 범용적인 질문을 던져 면접 흐름이 끊기지 않도록 방어합니다.
요약
RAG 활용 여부: Yes. 1단계 생성 시 RAG 기반의 질문 데이터를 우선적으로 참조합니다.
RAG 실패 대비 여부: Yes. RAG 기반 생성 실패 시 정적 질문 은행(QBank) → 긴급 질문 리스트(Emergency) 순으로 자동 전환되어 면접 세션은 어떤 상황에서도 중단 없이 진행되도록 설계되어 있습니다.

Q10 : '면접 세션 엔진의 세션상태, 페이즈 흐름, 단계 전이 로직, 꼬리 질문 생성, 질문 개수 종료 처리가 잘 구현되어있는가?'

네, 면접 세션 엔진(InterviewSessionEngine)은 말씀하신 핵심 기능들이 체계적인 상태 기계(State Machine) 구조로 견고하게 구현되어 있습니다.
주요 구현 세부 사항은 다음과 같습니다:
1. 세션 상태 및 단계 전이 로직
상태 정의: APPLIED(지원됨), IN_PROGRESS(진행 중), COMPLETED(완료), INTERRUPTED(중단), ABORTED(실패)의 5가지 명확한 상태를 가집니다.
원자적 전이: 모든 상태 변경은 _atomic_commit()을 통해 PostgreSQL(권위 저장소)에 먼저 저장된 후 Redis(미러)에 반영되도록 설계되어 있어, 갑작스러운 장애 시에도 데이터 유실이나 상태 불일치가 발생하지 않습니다.
2. 페이즈 흐름 (PhaseManager)
구조화된 흐름: PhaseManager 클래스가 면접의 단계를 다음과 같이 엄격하게 순서대로 제어합니다.
OPENING: 면접 시작 (1문항)
MAIN: 직무 및 역량 질문 (설정된 개수만큼)
FOLLOW_UP: 답변에 따른 꼬리 질문 (선택적)
CLOSING: 면접 마무리 (1문항)
순서 보장: 현재 문항의 인덱스와 전체 문항 수를 계산하여 다음 단계가 무엇인지 실시간으로 판단합니다.
3. 꼬리 질문 생성 (Follow-up)
지능적 생성: FOLLOW_UP 단계에서는 LLM이 이전 답변 내용을 분석하여 깊이 있는 추가 질문을 생성합니다.
제한 로직: 무한정 꼬리 질문이 이어지지 않도록 tail_question_limit (현재 2회로 설정)을 두어 면접 시간을 조절합니다.
4. 질문 개수 및 종료 처리
정확한 카운팅: completed_questions_count가 설정된 total_question_limit에 도달하면 자동으로 terminate_session을 호출합니다.
런타임 검증 (Assertion): 세션 종료 시, 시스템은 다음 사항을 마지막으로 검증합니다.
MAIN 질문이 약속된 개수만큼 수행되었는가?
시작(OPENING)과 끝(CLOSING)이 올바른 단계로 끝났는가?
만약 이 조건이 맞지 않으면 PhaseAssertion 경고를 발생시켜 데이터 무결성을 유지합니다.
결론
현재 해당 엔진은 단순한 질문-답변 반복을 넘어, 면접의 처음부터 끝까지를 제어하는 정교한 파이프라인을 갖추고 있습니다. 
특히 WiringFlags를 통해 각 기능의 활성화 여부를 유연하게 제어할 수 있어 확장성까지 고려된 상태입니다. (packages/imh_session/engine.py 및 phase_manager.py 참조)

Q11 : '질문에 대한 답변은 어떻게 평가되고 있는가?'

질문에 대한 답변은 루브릭(Rubric) 기반의 4단계 정량 평가 시스템을 통해 객관적으로 채점되고 있습니다. 모든 평가는 1점~5점 척도로 산출되며, 최종 점수는 각 항목에 설정된 가중치에 따라 합산됩니다.
상세 평가 항목은 다음과 같습니다:
1. 직무 역량 (Knowledge)
평가 기준: 답변 내 업무 관련 핵심 키워드의 포함 여부 및 기술적 복잡도를 분석합니다.
점수 산출: 키워드 매칭 개수가 4개 이상이면 5점, 1개면 2점 등을 부여하며, 답변이 지나치게 단순할 경우 점수가 하향 조정될 수 있습니다.
2. 문제 해결력 (Problem Solving)
평가 기준: 면접 진행 중 AI가 제공한 힌트(Hint)의 사용 횟수를 측정합니다.
점수 산출: 힌트 없이 스스로 해결했을 경우 5점을 부여하며, 힌트 사용 횟수가 늘어날수록 점수가 차감됩니다 (4회 이상 사용 시 1점).
3. 의사소통 능력 (Communication)
평가 기준: 후보자의 답변이 STAR 구조(Situation, Task, Action, Result)를 갖추고 있는지 분석합니다.
점수 산출: 논리적인 STAR 구조가 감지되면 5점을 부여하고, 그렇지 않은 경우 평균 점수인 3점을 기본으로 부여합니다.
4. 태도 및 비언어 분석 (Attitude)
평가 기준: 멀티모달 엔진을 통해 수집된 **시선 처리(Gaze)**와 **감정 상태(Emotion)**를 분석합니다.
점수 산출:
시선이 정면(카메라)을 80% 이상 응시하면 만점.
공포, 슬픔 등 부정적 감정 표현 비율이 10% 미만이면 좋은 점수를 부여합니다.
최종 점수 합산 및 결과 저장
이렇게 산출된 4가지 영역의 점수는 공고(JobPolicy)에 설정된 가중치(예: 직무 40%, 소통 30% 등)에 따라 최종 합산됩니다.
평가 결과는 PostgreSQL에 영속적으로 저장되어 관리자가 즉시 확인할 수 있으며, 동일한 입력에 대해 항상 동일한 결과가 나오도록 **결정적 해시(Input Hash)**를 생성하여 투명성을 보장합니다. 
(packages/imh_eval/engine.py 참조)

Q12 : 'STT, vision 감정분석, 음성pitch와 volume 등 멀티모달의 데이터가 면접 평가에 잘 반영될 수 있나?'

네, STT와 시각/음성 분석을 포함한 멀티모달 데이터는 이미 시스템 내부에 수집 파이프라인이 구축되어 있으며, 일부 핵심 지표는 실제 평가 점수에 즉시 반영되고 있습니다.
상세 반영 현황은 다음과 같습니다:
1. 시각 및 감정 분석 (Vision & Emotion) -> [반영 중]
시선 처리(Gaze): MediaPipe를 통해 면접자의 시선이 카메라 정면을 향하는지 실시간 측정하며, 이 데이터는 '태도/비언어' 항목 점수의 약 50%를 결정합니다.
감정 분석(Emotion): DeepFace 엔진을 통해 면접 중 감정 변화를 추적하며, 공포나 슬픔 등 부정적 감정의 빈도가 높을 경우 '태도' 점수에 감점 요인으로 반영됩니다.
2. 음성 분석 (Pitch & Intensity) -> [수집 중 / 반영 예정]
데이터 추출: Parselmouth 엔진을 통해 목소리의 높낮이(Pitch)와 성량(Intensity/Volume)을 2.5초 단위로 정밀하게 추출하고 있습니다.
현재 상태: 이 데이터는 현재 데이터베이스(multimodal_observations)와 실시간 결과 화면(Projection)에는 기록되고 있으나, 최종 루브릭 점수 산출 로직에는 아직 직접적인 가중치가 할당되지 않은 '수집 및 관측' 단계입니다. (향후 자신감 점수 등에 활용 가능)
3. STT (Speech To Text) -> [핵심 반영 중]
내용 분석: Faster-Whisper를 통해 변환된 텍스트는 '직무 역량' 평가의 핵심 데이터로 사용됩니다. 답변 내 기술 키워드 포함 여부와 답변의 논리적 구조(STAR 구조)를 파악하는 데 직접적으로 활용됩니다.
요약 및 한계
반영 수준: 현재 **시선(Gaze)**과 감정(Emotion), **STT(Text)**는 실제 점수에 직접 영향을 미칩니다.
기술적 준비: 음성 피치와 볼륨은 데이터로서 완벽히 저장되고 있어, 평가 엔진(RubricEvaluator)의 가중치 설정만 조정하면 즉시 점수화가 가능한 상태로 준비되어 있습니다.
결론적으로, 멀티모달 데이터는 단순 기록을 넘어 면접자의 태도와 답변의 진실성을 입체적으로 평가하는 근거로 활용되고 있습니다. (packages/imh_multimodal 및 imh_eval 모듈 참조)

Q13 : '세션 조회 시 Redis → PostgreSQL fallback 구조인가?'

네, 맞습니다. 
InterviewSessionEngine의 세션 로드 로직(_load_or_initialize_context)을 보면 **Hot Storage(Redis/Memory)**에서 상태를 먼저 조회합니다.
여기서 조회가 실패(Miss)하면 **PostgreSQL(Authority)**에서 데이터를 읽어와 복구하는 Hydration(수분 공급) 프로세스가 가동됩니다. 이를 통해 메모리 데이터가 휘발되더라도 PostgreSQL에 저장된 유일한 권위(Single Source of Truth)를 통해 세션이 완벽히 복원됩니다.
__________________________________________________

Q14 : '한 세션에 동시에 두 API 호출 시 질문이 두 개 생성되는가?'

아니요, 그렇지 않습니다. 
SessionService의 모든 주요 쓰기 작업(submit_answer 등)은 **Redis 기반 분산 락(ConcurrencyManager)**을 통해 보호됩니다.
동일 세션에 대해 동시에 요청이 오면, 하나의 요청만 락을 획득하여 정상 처리합니다.
다른 요청은 Fail-Fast 원칙에 따라 즉시 차단(423 Locked)되므로, 중복 질문 생성과 같은 레이스 컨디션(Race Condition)이 원천적으로 방지됩니다.
__________________________________________________

Q15 : '동일 answer submit API를 두 번 보내면 질문이 두 번 생성되는가?'

아니요, 중복 생성되지 않습니다. 시스템은 request_id 기반의 멱등성(Idempotency) 보장 로직을 갖추고 있습니다.
첫 번째 요청이 처리되면 해당 결과를 Redis에 캐싱합니다.
동일한 request_id로 재요청이 오면, 엔진을 다시 구동하지 않고 캐싱된 기존 응답값을 즉시 반환(HTTP 200)합니다.
__________________________________________________

Q16 : '프론트가 면접 상태를 polling 없이 읽을 수 있는가? (예: 현재질문, 남은질문 수, 세션상태)'

네, 가능합니다. 시스템은 SSE(Server-Sent Events) 기술을 사용하여 실시간 프로젝션 스트림을 제공합니다 (/sessions/{session_id}/multimodal/stream).
서버에서 상태가 변경되면(질문 생성, 점수 산출 등) Redis Pub/Sub을 통해 프론트엔드로 즉시 이벤트를 푸시합니다.
프론트엔드는 useSSEProjection 훅을 통해 Polling 없이도 현재 질문, 남은 질문 수, 면접 상태를 실시간으로 업데이트 받아 화면에 반영할 수 있습니다.
__________________________________________________

Q17 : '관리자기능 1. 공고 생성 가능, 2. 공고 게시 가능, 3. 공고 종료 가능, 4. 공고 수정 가능, 5. 통계 조회 가능'

공고 생성/게시/종료 (1~3):
IMH/api/jobs.py 내의 POST /jobs, /publish, /close 엔드포인트를 통해 실제 작업이 수행됩니다.
Job 엔티티(packages/imh_job/models.py)가 상태 전이 로직(DRAFT → PUBLISHED → CLOSED)을 엄격히 관리하며, 게시 시 정책 데이터가 스냅샷으로 **박제(Immutability)**됩니다.
공고 수정 (4):
Job.update_policy 로직을 통해 DRAFT 상태에서는 정책 수정이 가능하나, PUBLISHED 이후에는 AI 평가의 공정성을 위해 메타데이터(모집 인원 등)만 수정 가능하도록 제한되어 있습니다.
통계 조회 (5):
IMH/api/admin.py 의 통계 API가 구현되어 있으며, PostgreSQL의 원본 세션 데이터를 기반으로 합격/불합격률 및 일별 통계를 실시간 집계(Phase 4)하여 반환합니다.

Q18 : '면접자기능 1. 공고 목록 조회, 2. 공고 지원, 3. 면접 시작, 4. 질문 응답, 5. 면접 결과 조회'

공고 목록 조회/지원 (1~2):
지원자가 PUBLISHED 상태의 공고만 조회하고 지원할 수 있는 필터링 로직이 적용되어 있습니다.
POST /sessions 호출 시, 지원 정보를 바탕으로 새로운 면접 세션이 DB에 생성됩니다.
면접 시작 (3):
InterviewSessionEngine.start_session()이 호출되면 상태가 IN_PROGRESS로 변하며, 첫 번째 질문(OPENING)이 생성되어 DB 및 Redis에 기록됩니다.
질문 응답 (4):
submit_answer API를 통해 답변이 제출되면, 세션 엔진이 응답을 기록하고 다음 질문을 생성(RAG/LLM)하거나 종료 페이즈로 전이하는 루프 동기화가 구현되어 있습니다.
면접 결과 조회 (5):
GET /interviews/{id}/result 엔드포인트를 통해 채점이 완료된(EVALUATED) 세션의 항목별 점수와 총평을 조회할 수 있으며, 이는 evaluation_scores 테이블의 실제 점수 데이터를 기반으로 합니다.

Q22 : 'OPENING / MAIN / FOLLOW_UP / CLOSING 단계에 따라 질문 스타일이 달라지는가'

MAIN: 직무 및 역량 중심의 심층 질문 (이력서 컨텍스트 주입).

CLOSING: 입사 후 포부 및 마무리 멘트.

메커니즘: imh_qbank의 벡터 DB에서 현재 지원 직무와 유사한 질문 후보들을 검색하여 LLM이 참고할 수 있도록 컨텍스트를 제공합니다.

Q24 : 'LLM 질문 생성 실패 시 다른 방식으로 질문이 생성되는가'

반영 여부: Yes. 시스템은 3단계 계층형 방어 구조를 가집니다.
Tier 1: RAG + LLM 동적 생성.
Tier 2 (Fallback): LLM 생성 실패 시, 미리 정의된 **정적 질문 은행(Static QBank)**에서 임의의 질문을 추출.
Tier 3 (Emergency): 질문 은행마저 고갈될 경우를 대비한 비상 범용 질문 리스트(예: "본인의 강점을 소개해 주세요")를 즉시 반환하여 면접 중단을 원천 차단합니다.

Q33 : '면접 상태가 정상적으로 전이되는가'

Yes.

APPLIED → IN_PROGRESS → COMPLETED → EVALUATED

Q34 : '동일 세션 중복 실행 방지 기능이 있는가'

Yes. 두 가지 수준의 방어막이 구축되어 있습니다.
동시성 제어(Redis Lock): 동일 세션에 대해 동시에 여러 API 요청이 들어올 경우, Redis 분산 락을 사용하여 하나의 요청만 처리하고 나머지는 차단합니다.
멱등성(Idempotency) 보장: 동일한 요청(request_id)이 재전송될 경우, 작업을 다시 수행하지 않고 기존에 저장된 응답을 즉시 반환하여 중복 처리를 방지합니다.

Q35 : 'ResumeSummarizer가 이력서의 정보를 LLM에 전달한다고 되어있는데, 공고와 다르게 이력서는 하드코딩되어있지 않은 상태인가?'

1. ResumeSummarizer는 하드코딩인가?
아니요, 하드코딩되어 있지 않습니다.
packages/imh_providers/resume_summarizer.py를 보면, 업로드된 PDF/TXT의 실제 텍스트(raw_text)를 LLM에 전달하여 실시간으로 요약을 생성하는 로직이 구현되어 있습니다.
단, 텍스트 추출에 실패하거나 이력서가 없는 경우를 대비한 **GENERIC_FALLBACK_SUMMARY**라는 하드코딩된 문구가 존재하며, 현재 연결이 불안정한 경우 이 문구가 대신 사용될 수 있습니다.
2. 실제 데이터 전달의 "미완성" 상태
분석 결과, 현재 시스템에는 다음과 같은 '연결 고리'의 공백이 존재합니다.
저장 위치의 분리: 이력서 요약은 resumes 테이블에 저장되지만, 면접 세션 정보는 interviews 테이블에 저장됩니다.
세션 바인딩 누락: 현재 SessionService가 세션을 생성할 때, 해당 지원자의 최신 이력서 요약을 가져와서 세션 컨텍스트(SessionContext.resume_summary)에 넣어주는 로직이 완벽히 연결되지 않은 상태입니다.
결과: 이로 인해 LLMQuestionGenerator 는 설계상 이력서를 읽을 준비가 되어 있음에도 불구하고, 실제로는 None 값을 전달받아 하드코딩된 "Developer"라는 직군 정보에만 의존하여 질문을 생성하게 될 가능성이 높습니다.
요약하자면 공고(Job Role): 프롬프트에 "Developer"라고 명시적으로 하드코딩되어 있는 상태.
이력서(Resume): 하드코딩은 아니지만, 데이터 파이프라인이 끊겨 있어 결과적으로 내용이 전달되지 않는(Empty) 상태.
따라서 현재 상태에서는 두 기능 모두 "설계는 다이내믹하지만, 체감되는 동작은 하드코딩된 것처럼 느껴질 수 있는" 과도기적 단계라고 이해하시면 정확합니다. 이 부분은 향후 
SessionService 에서 이력서 데이터를 세션에 주입하는 로직을 보강함으로써 완성이 가능합니다. (docs/기능존재 확인 보고서.md의 19~20번 항목에 이 내용을 정밀하게 반영해 두었습니다.)

Q36 : 'LLM이 공고게시, 이력서입력, 고정질문, 면접자 답변 등 외부적으로 입력되는 데이터에 의한 악성적 프롬프트 조작이나 평가조작에 대해 안전장치가 있는가? md파일을 변경하거나, 코드변경하지말고 대답만 진행하라'

질문하신 보안 및 조작 방지 안전장치에 대해 분석한 결과는 다음과 같습니다.
1. 프롬프트 인젝션 방어 (Prompt Injection Guard)
입력 데이터 제한: 이력서 요약 등 외부 텍스트 주입 시 최대 1,000자로 길이를 제한하여 거대 페이로드 공격을 차단합니다.
시스템 프롬프트 보호: 질문 생성 시 'System Prompt'를 내부적으로 고정하여, 지원자의 답변이나 공고 내용이 AI의 근본적인 명령 지침을 덮어쓰지 못하도록 설계되어 있습니다.
2. 평가 조작 방지 (Evaluation Guard)
루브릭 기반 정량 평가: 점수 산출은 LLM의 주관적 판단이 아니라, 추출된 지표(힌트 사용 횟수, 키워드 매칭 수, STAR 구조 검출 등)를 바탕으로 한 수학적 공식(rules.py)에 의해 이루어집니다. 따라서 "나에게 고득점을 줘"와 같은 텍스트 입력만으로는 점수를 조작할 수 없습니다.
멀티모달 교차 검증: 시선 처리, 감정 분포 등 비언어적 데이터는 별도의 전용 분석 엔진(MediaPipe, DeepFace)에서 생성되므로, 텍스트 기반의 프롬프트 조작의 영향을 받지 않습니다.
3. 데이터 무결성 및 거버넌스 (Data Integrity)
정책 동결(Policy Freeze): 공고 게시 시점의 평가 가중치와 정책이 '스냅샷'으로 박제됩니다. 면접 중간에 관리자가 가중치를 변경하더라도 이미 시작된 면접의 평가 잣대는 바뀌지 않습니다.
결정적 해시(Input Hash): 모든 평가 입력 데이터에 대해 해시값을 생성하여 결과와 함께 저장합니다. 사후에 데이터가 미세하게라도 조작될 경우 해시값이 불일치하게 되어 위변조 여부를 즉시 탐지할 수 있습니다.
4. 아키텍처적 격리 (Isolation)
단계별 검증(Phase Manager): 질문 생성과 답변 제출은 면접 단계 관리자의 엄격한 시퀀스 통제를 받습니다. 악성 입력을 통해 단계를 건너뛰거나 비정상적으로 면접을 종료시키려는 시도는 엔진 레벨에서 차단됩니다.
