# [TASK-033] 모델 비교 및 평가 취약점 탐지 테스트 계획

본 계획은 [Case A] Backend Developer 공고를 기반으로 4가지 상황을 변형하여 메인/Fallback LLM을 평가하고, AI 면접 및 평가 프롬프트의 취약점을 탐지하기 위한 통합 테스트 파이프라인 설계안입니다.
**엔진 구조, DB 스키마 분리, Authority/Snapshot 계약 등 기존 아키텍처는 절대 변경하지 않으며**, 프롬프트 제안, 모델 라우팅 정책, Test Harness 및 리포트 설계 영역에서만 보완점을 제안합니다.

---

## 1. 4상황(S1~S4) 변형 및 생성 규칙 (유지 + 철학 명시)

베이스 이력서 1개를 바탕으로 **축 A(직무 적합성)**와 **축 B(답변 품질)**에 따라 4가지 테스트 프로필을 생성합니다.
(공고의 직무/기술 스택은 원본 유지)

### ⚠️ 평가 철학 
- **이력서는 질문 방향을 정하기 위한 단순 참고 자료입니다.**
- **후보자의 직무 역량(적합성)은 반드시 면접 과정(질문/답변)에서 검증되어야 합니다.**
- **따라서 이력서상 불명확한 부분에 대한 편향적 감점을 절대 금지합니다.**

### 조합된 4상황 시나리오
- **[S1] A1+B1 (적합+우수)**: 이력서와 답변 모두 우수. 고득점 도출 기대.
- **[S2] A1+B2 (적합+불량 위장)**: 이력서는 훌륭하나, 답변이 모순적임. (과장 탐지 및 드릴다운 테스트)
- **[S3] A2+B1 (부적합+우수)**: 이력서상 경험은 불명확하나, 실무에 대한 답변 역량이 우수함. (이력서 편향 없는 정당한 평가/진주 찾기 테스트)
- **[S4] A2+B2 (부적합+불량 위장)**: 이력서 불명확, 답변 불량. 정당화된 감점 기대.

### 불량 답변(B2) 위장 패턴
- **STAR 구조 흉내**: "문제->행동->결과" 포맷을 띄지만, 'How(어떻게 통과/개선했는가)'에 대한 기술적 논리가 공백임.
- **책임 모순**: 주도적 설계(100%)에서 단순 유지보수 참오(API만 연결)로 꼬리질문 시 입장이 바뀜.
- **기술 빙고**: 질문과 무관한 기술 스택(MSA, K8s, Redis, Kafka 등)을 인과관계 없이 나열함.

## 2. 면접 흐름 취약점 탐지 보강 (Drill-down 강화)

### Surface Acceptance Error 플래그
면접관(LLM)이 B2 위장 답변을 들었을 때 표면적 논리만 수긍하고 넘어가면 오류 플래그로 집계합니다.
다음 중 **단 1개도 수행하지 않고 칭찬/다음 질문으로 넘어가면 플래그 처리**합니다:
1. 구체적인 수치 재질문 (예: "결제 지연을 90% 줄인 핵심 최적화 요인 한 가지는 무엇인가요?")
2. 역할 범위 재확인 (예: "직접 설계하신 부분과 인프라팀이 구성해준 부분의 경계는 어디입니까?")
3. Why(선택 이유) 재검증 (예: "Kafka가 아닌 다른 큐나 동기식 처리를 고려한 적은 없습니까?")
4. 실패/장애 경험 검증 (예: "해당 구조 도입 시 겪었던 가장 큰 기술적 트러블슈팅 사례를 말씀해주세요.")

*※ 이 플래그 발생 횟수 및 비율은 모델별 통계로 기록됩니다.*

## 3. LLM 자체 성능 분리 테스트 (Engine 독립)

면접 시나리오 흐름과 별개로 LLM의 순수 인지/판단 역량을 독립적으로 검증합니다.

### 3.1 Direct Reasoning Test
- **입력**: B2 방식으로 생성된 위장 답변 텍스트.
- **Direct Prompt**: "다음 답변의 기술적 모순, 과장, 역할 불일치, 인과 오류를 모두 지적하라."
- **수집/측정 항목**:
  - 탐지 성공 여부 (결함 탐지율)
  - 지적 개수 및 분석의 깊이 (정확성)
  - 논리적 일관성 확보 여부

### 3.2 Strict JSON Contract Test
- **실행**: 50회 연속 독립 호출 (온도/파라미터 고정).
- **조건**: 답변을 오직 Strict JSON 형식으로만 반환할 것을 강제.
- **수집/측정 항목**:
  - JSON 파싱 실패율 (%)
  - 불필요 텍스트(마크다운 코드블록 이외의 서문/결문) 혼입 비율 (%)

## 4. Output Pollution / 문자열 오염 테스트

면접/평가 과정 중 모델의 응답에 부작용 문자가 포함되는 빈도와 유형을 테스트합니다.

### 4.1 오염 유형 및 등급
- **유형 A (Sev 1)**: 한국어/영어 이외의 무의미한 문자/언어 혼입 (진행방해 없음)
- **유형 B (Sev 2)**: 깨진 유니코드, 랜덤 문자 출력 (이해/렌더링 문제)
- **유형 C (Sev 2)**: 이스케이프/제어 문자 오류 (UI/프론트엔드 파손 유발)
- **유형 D (Sev 3 - 즉시 탈락)**: JSON 파손(문법 오류, 키값 불일치, 하프 생성) 등 엔진 에러 유발

### 4.2 측정 항목
- 전체 Pollution Rate (%)
- Sev 등급별 발생 비율 파악

---

## 5. Hallucination Trap (환각 유도 테스트)

공고나 이력서에 존재하지 않는 허위/혼종 기술을 후보자가 답변에 의도적으로 삽입하여 면접관의 반응을 살핍니다.
- **예시 삽입구**: "제가 겪은 장애를 Google Spanner 기반 CQRS를 MongoDB Redis 트랜잭션으로 엮어 해결했습니다." (기술적으로 과도하거나 모순된 스택 조합)
- **측정 항목**:
  1. 면접관 LLM이 이 헛소리를 감지하고 **상세 검증 질문을 던지는가?**
  2. 최종 평가 시 **이를 의심 없이 수용하여 기술 점수를 부여(과대평가)하는가?**

---

## 6. 메인 / Fallback 선정 기준 (정량화)

테스트 결과를 바탕으로 실서비스에 투입할 모델 등급을 부여합니다.

### Main 모델 (핵심 운영자격)
- Direct Reasoning Test 성공률 **≥ 80%**
- B2 지원자 과대평가율(착시 비율) **≤ 10%**
- Surface Acceptance Error (Drill-down 실패율) **≤ 15%**
- JSON 파싱 Error **≤ 1%**
- Pollution Sev3 발생 횟수 **= 0** (제로 톨러런스)
- P95 Latency **≤ (엔진 요구 기준값, 예: 3.5초)**

### Fallback 모델 (백업/안정성 자격)
- JSON 응답 안정성 우수 (에러율 1% 미만, 복구용이)
- Latency 안정성 및 처리율 우수 (가변폭 적음)
- Reasoning 역량 **중간 수준 이상** (Main 대비 섬세함이 부족하더라도 다운타임을 방어할 수 있는 최소 논리력)

---

## 7. 리포트 산출 포맷 및 계층 (Layer)

최종 평가 리포트는 각 LLM마다 아래 **5개 레이어(L1~L5) 점수**를 산출하여 매트릭스 형태로 제공됩니다.

- **[L1] LLM 순수 성능 (Pure Reasoning)**: Direct Reasoning 탐지율, 심도.
- **[L2] 면접 흐름 검증 능력 (Interview Control)**: 긴장감 형성 여부, Surface Acceptance Error 방어율, Hallucination Trap 탐지율.
- **[L3] 평가 루브릭 판별력 (Scoring Accuracy)**: A2 이력서 편향 배제, S2(B2) 과대평가 방지(가짜 높은 점수 억제율).
- **[L4] 구조 안정성 (Contract & Pollution)**: JSON Contract 성공률, 문자열 오염(Pollution) 빈도 및 Sev3 결함 제로 여부.
- **[L5] 운영성 (Latency & Robustness)**: P95/P99 응답 속도, Timeout 비율 등 퍼포먼스 메트릭.

이 리포트를 바탕으로 최종 LLM 라우팅 설정값(Main / Fallback 지정 및 Timeout/Retry 정책)과, 엔진 안정성을 확보할 수 있는 시스템 프롬프트(지시문) 보완책을 제시합니다.
