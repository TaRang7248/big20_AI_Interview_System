# TASK-027 / CP2 Plan: RAG Cache & Cost Optimization

## 1. Purpose
The primary goal of CP2 is to introduce a **RAG Result Cache** layer using Redis to optimize **Cost** (LLM Token Usage) and **Latency** (Response Time).
- **Optimization Only**: This cache is strictly for read optimization and is **NOT** an Authority.
- **Graceful Degradation**: The system must function correctly (via PostgreSQL + RAG Pipeline) even if Redis is down or the cache is empty.
- **No Persistence Dependency**: Redis is treated as ephemeral; PostgreSQL remains the sole source of truth for all persistent data.

## 2. Scope & Out of Scope

### IN SCOPE
- **RAG Result Cache Definition**: Caching of final answers, evi, context summaries, and metadata generated by the RAG pipeline.
- **Key Design**: Composite keys including Job ID, Job Policy Snapshot, Question Intent, User Input Hash, and Configuration Version.
- **Read-Through Strategy**: Standard Cache-Aside / Read-Through pattern with **Asynchronous Save**.
- **Cache Invalidation Policy**: 
    - Triggered by Authority changes (e.g., Job Policy update, Question Bank update).
    - "Invalidate First" or "Versioned Key" strategy selection.
- **TTL & Eviction Policy**: 
    - Base TTL definition (24h default).
    - Dynamic TTL based on traffic/mode.
    - Max Memory / Eviction Policy (`volatile-lru`).
- **Redis Down Handling**: Circuit Breaker / Fallback to direct RAG calculation without error.
- **Authority Limits**: Explicit prohibitions on using Cache for Engine decisions.
- **Approval Gates**: Criteria for moving to Implementation.

### OUT OF SCOPE
- **CP1 Projection Logic Changes**: CP1 is LOCKED.
- **Write-Back Cache**: No data flows from Redis to PostgreSQL.
- **PostgreSQL Schema Changes**: No structural changes to the DB.
- **Snapshot Contract Changes**: Job/Session Snapshot logic remains immutable.
- **Engine/Domain Policy Changes**: Core session logic is untouched.
- **Candidate Cache (CP3) & Prompt Cache (CP4)**: Reserved for future phases.
- **Model/Prompt Quality Improvements**: Focus is strictly on caching existing outputs.

## 3. PostgreSQL vs Redis Responsibility Boundaries
- **PostgreSQL (Sole Authority)**:
    - Stores all persistent state, history, and configuration.
    - Is the **only** source of truth for Job Policies, Session States, and Question Bank data.
    - Decisions on "Correctness" are derived solely from PG data.
- **Redis (Optimization & Ephemeral)**:
    - Stores **derived** RAG results to save computation.
    - Holds **no** unique data; everything in Redis can be reconstructed from PG + RAG Pipeline.
    - **No Write-Back**: Data **never** flows from Redis to PostgreSQL.
    - **Loss Tolerance**: Redis flush or failure has **zero** impact on data integrity, only performance.

## 4. Cache Authority Limits
To ensure system stability and correctness, the following limits are enforced:
- **No Engine Decisioning**: RAG Cache results must **NOT** be used for Engine state transitions, Policy enforcement, or Evaluation logic.
- **No Truth Claim**: The Cache contains "Previous Calculation Results," not "The Truth."
- **No Admin Reliance**: Admin Query Layer must continue to rely on PostgreSQL (or standard Repositories) for accurate reporting.

## 5. Read-Through Strategy
The RAG retrieval process will follow this flow:
1.  **Redis Lookup**: Query Redis with the specific Cache Key.
2.  **Hit**:
    -   Deserialize result.
    -   Return cached RAG result.
3.  **Miss (or Error)**:
    -   **Log** (if Error): Log Redis timeout/connection error as Warning.
    -   **Calculate**: Execute the standard RAG Pipeline (Retrieve Context -> LLM Generation).
    -   **Save (Policy: Asynchronous Background Task)**:
        -   **Decision**: Fire-and-forget background task (or non-blocking async call).
        -   **Reason**: User Latency is paramount. The RAG calculation is already slow; adding synchronous Redis Write latency (even if small) is unnecessary for an optimization layer.
    -   **Return**: Return the detailed RAG result immediately.
4.  **Failure Safety**: If saving to Redis fails (in background), log the error. The user is already served, so no impact.

## 6. Key Input Components
To ensure cache correctness and avoid collisions (returning wrong answers for different contexts), the Cache Key is strictly defined.

### 6.1. User Input Normalization
Before hashing, user input (Question/Query) must be normalized:
-   **Trim**: Remove leading/trailing whitespace.
-   **Collapse Whitespace**: Replace multiple spaces/tabs with single space.
-   **Normalize Line Breaks**: Replace all CRLF/CR with LF.
-   **Case Sensitivity**: **Preserve Case**. RAG relies on semantic nuance (e.g., "Python" vs "python", acronyms). Aggressive lowercasing might lose meaning in vector space.

### 6.2. Hash Components (Ordered List)
The Cache Key is a specific hash (e.g., SHA-256) of the following concatenated components:

**A. Essential (MVP Key)**
1.  `job_id`: Partition by Job Posting.
2.  `job_policy_version`: Snapshot version of the Job Policy (if retrieval depends on Policy).
3.  `normalized_user_input`: The actual query text.
4.  `model_name`: "gpt-4" vs "gpt-3.5" (Results differ by model).
5.  `prompt_template_version`: If the system prompt changes, previous answers are invalid.

**B. Extended (Future/Full Key)**
6.  `generation_params`: `temperature`, `top_p`. (Different params = different result).
7.  `retrieval_params`: `top_k`. (Different context = different result).
8.  `rag_index_version`: Version of the Vector DB / Question Bank (Content drift).

### 6.3. Key Construction Principle
-   **Rule**: `Hash(Component_A + Component_B + ...)`
-   **Collision Safety**: If *any* component that affects the output changes, the Key *must* be different.

## 7. Metrics & Targets (Cost Optimization)
CP2 success is measured by the following KPIs. These will be recorded via logs/metrics.

### 7.1. Defined Metrics
-   `cache_hit_ratio`: `(Total Hits / Total Requests) * 100`.
-   `rag_invocations_saved`: Count of RAG calls avoided.
-   `token_usage_saved`: Estimated tokens saved based on avg tokens per RAG call.
-   `p95_latency_delta`: `p95(Miss Latency) - p95(Hit Latency)`.

### 7.2. Initial Targets (Hypothesis)
-   `cache_hit_ratio`: > **15%** (Conservative start, assuming some repeated questions/retries).
-   `latency_reduction`: Hit should be < **50ms** (vs RAG 2-5s).

## 8. TTL & Eviction Policy
### 8.1. Default TTL: 24 Hours
-   **Reasoning**:
    -   **Daily Cycle**: Interview traffic often follows daily patterns. 24h covers global timezone active windows.
    -   **Stability**: Configs/Prompts don't change hourly. 24h is a safe upper bound to auto-correct "Zombie Keys" if granular invalidation misses something.
    -   **Cost/Freshness Balance**: Re-computing a query once a day is an acceptable cost for ensuring freshness.

### 8.2. Dynamic TTL Rules
-   **High Traffic Jobs** (`active_candidates > 100`): **48 Hours**. (Higher hit probability).
-   **Debug/Dev Mode**: **1 Hour** (or 0). Rapid iteration requires fresh results.
-   **Model Cost**: (Optional) Expensive Models (e.g., GPT-4) -> **48 Hours** to maximize savings.

### 8.3. Eviction Policy
-   **Redis Config**: `maxmemory-policy volatile-lru`
-   **Behavior**: Evict keys with an implementation of TTL that are Least Recently Used.
-   **Safety**: Allows the cache to fill up without crashing Redis; just drops old cached items.

## 9. Redis Down Policy
-   **Principle**: Redis is an **Auxiliary Service**.
-   **Behavior**:
    -   **Connection Error / Timeout**: Catch exception immediately.
    -   **Action**: Log `WARNING` ("Redis RAG Cache unavailable, bypassing...").
    -   **Fallback**: Proceed directly to RAG Pipeline (PostgreSQL + LLM).
    -   **No Crash**: The application must **never** Raise 500 due to Redis.
